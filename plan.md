## Q1 范围说明

**本次迭代目标：** 实现一轮对话的基础功能验证，打通“用户提问 → 检索 → 生成答案 → 展示引用”的完整链路。

**实现的模块：**

- 用户层：Web 聊天页面（流式输出、源文档引用）
- Agent 层：简单的推理引擎（单步调用检索工具 → 生成最终答案）
- 工具集：检索工具（封装混合检索）
- 数据处理管线：本地文档 → 解析 → 分块 → 向量化 → 写入存储
- 数据持久化层：向量存储、文档存储、元数据存储

## 二、各模块详细目标

### 模块 1：用户层

## 一、模块说明

用户层负责提供 AI 智能问答系统的 Web 交互入口。用户在页面输入自然语言问题后，前端将问题发送给 Agent 层，接收并展示 Agent 返回的答案、引用来源和异常状态。

Q1 阶段用户层的核心目标是打通基础问答交互链路：

> 用户输入问题 → 调用 Agent 接口 → 展示答案 → 展示引用来源 → 异常时给出提示

用户层不负责检索、Prompt 组装、LLM 调用、文档解析和向量存储，这些由 Agent 层、工具集、数据处理管线和数据持久化层完成。

Q1 建议技术方案：

-  前端：Vue 
-  用户层后端 / 路由转发：Flask，可选 
-  Agent 调用方式：HTTP / SSE / fetch stream 
-  Mock：本地 Mock 数据或 Mock API

## 模块 2：Agent 层 

### 一、模块概述

Agent 层负责接收用户问题，调用检索工具获取相关文档块，将用户问题与检索结果组装成 Prompt，调用大语言模型生成最终答案，并返回带引用来源的回答。

Q1 阶段只实现 **简化版单轮 RAG Agent**，目标是打通基础问答链路，不实现复杂 Agent 能力。

本阶段 Agent 层的核心目标是：

用户输入一个问题后，Agent 能够调用一次检索工具，获取相关文档块，基于这些文档块生成答案，并返回自然语言答案和可供前端展示的引用信息。

### 二、模块输入与输出

Agent 层输入：

- 用户原始问题 query 
- 可选检索参数，例如 top_k 
- 可选过滤条件，例如文档类型、空间、标签等。

Agent 层输出：

- 自然语言答案 
- 本次请求的 trace_id 
- 状态信息，例如成功、无相关文档、检索失败、模型调用失败 

建议返回结构：

{

  "trace_id": "string",

  "answer": "基于检索内容生成的自然语言答案，正文中可以使用 [1] [2] 形式标注引用。",

  "status": "success"

}

### 三、详细功能点清单

### 功能点 2.1：Agent 请求入口

功能描述：

Agent 需要提供一个统一入口，接收用户问题，并启动一次完整的问答流程。

子功能点：

- 接收用户问题 query 
- 判断问题是否为空 
- 为每次请求生成 trace_id 
- 将问题传入后续检索流程 
- 返回统一格式的响应结果 

验收标准：

- 输入正常问题时，Agent 能进入完整处理流程 
- 输入空字符串、纯空格、null 时，Agent 不调用检索工具，直接返回参数错误（参数校验）
- 每次请求都能生成唯一 trace_id 
- 日志中能看到该 trace_id 对应的一次完整调用过程：

参数校验（非空、长度限制）

生成 trace_id（UUID + 时间戳）

记录请求开始日志

调用检索工具

组装 Prompt

调用 LLM

格式化返回结果

记录完成日志

建议状态：

- Q1 必须完成 

### 功能点 2.2：检索工具调用

功能描述：

Agent 接收到问题后，必须调用检索工具获取相关文档块。Q1 阶段先实现调用一次检索工具。

子功能点：

- 将用户原始问题传给检索工具 
- 支持设置默认 top_k，建议为 3 到 5 
- 接收检索工具返回的文档块列表 
- 检索结果至少包含：文本内容、来源标题、相关度分数 
- 记录检索耗时、返回数量、最高相关度分数 

验收标准：

- 检索工具返回结果后，Agent 能正确读取文档块内容 
- 当检索工具返回 3 到 5 个文档块时，Agent 能继续进入 Prompt 组装流程 
- 日志中能看到本次检索的 trace_id、检索数量、检索耗时 
- 单元测试或 Mock 测试中能验证检索工具调用次数
- 打印检索返回的结果

建议状态：

- Q1 必须完成 

### 功能点 2.3：上下文组装器

功能描述：

上下文组装器负责把用户问题和检索结果组装成 LLM 可理解的 Prompt。Prompt 必须明确要求模型只能基于检索结果回答，不能编造知识库中不存在的信息。

子功能点：

- 组装系统指令 System Prompt 
- 组装用户问题 User Query 
- 组装检索上下文 Retrieved Context 
- 每个文档块需要包含来源标题、文档 ID、chunk ID、正文内容 
- Prompt 中需要明确要求模型使用引用编号回答 
- Prompt 中需要明确要求：如果检索结果不足以回答问题，应直接说明“当前知识库没有足够信息”，不得编造 

建议 Prompt 结构：

你是企业知识库问答助手。请严格基于给定的检索上下文回答用户问题。

要求：

1. 只能使用检索上下文中的信息回答。
2. 如果上下文中没有足够信息，请回答“当前知识库没有足够信息回答该问题”。
3. 不要编造文档中不存在的事实、数字、流程或结论。
4. 回答完成后，不要输出与上下文无关的内容。

用户问题：

{query}

检索上下文：

[1]

标题：{title}

doc_id：{doc_id}

chunk_id：{chunk_id}

内容：{chunk_text}

[2]

标题：{title}

doc_id：{doc_id}

chunk_id：{chunk_id}

内容：{chunk_text}

验收标准：

- Prompt 中必须包含用户原始问题 
- Prompt 中必须包含检索工具返回的文档块正文 
- Prompt 中必须包含每个文档块的来源信息 
- Prompt 中必须包含“不得编造”“仅基于上下文回答”的约束 
- 对同一个输入问题和同一组检索结果，组装出的 Prompt 结构稳定、可复现 
- 开发阶段能够打印或记录 Prompt，方便调试 

建议状态：

- Q1 必须完成 

### 功能点 2.4：LLM 推理引擎

功能描述：

推理引擎负责调用大语言模型，将组装后的 Prompt 发送给 LLM，并接收模型生成结果。

子功能点：

- 调用指定 LLM 接口 
- 支持设置模型名称、temperature、max_tokens 等参数 
- Q1 建议使用较低 temperature，例如 0.1 到 0.3，降低幻觉 
- 能接收 LLM 返回的答案文本 
- 能处理 LLM 调用失败、超时、返回为空等异常情况 
- 根据前端需求决定是否支持流式输出 

验收标准：

- 输入合法 Prompt 后，推理引擎能成功调用 LLM 
- LLM 返回内容不为空 
- LLM 返回内容与检索上下文相关 
- 当 LLM 接口失败时，Agent 能返回明确错误信息，而不是程序崩溃 
- 日志中能看到模型调用耗时、模型名称、trace_id 
- 如果 Q1 由 Agent 层负责流式输出，则需要验证前端能收到增量 token；如果流式输出由用户层包装，则 Agent 至少要能返回完整答案 

建议状态：

- Q1 必须完成 

### 功能点 2.5：答案生成

功能描述：

Agent 需要将 LLM 生成的自然语言答案和检索结果中的来源信息一起返回给前端，使前端能够展示答案和引用文档。

子功能点：

- 返回给前端answer

验收标准：

- 将返回给前端的answer直接打印出来

建议状态：

- Q1 必须完成 

### 功能点 2.6：基于检索结果回答，抑制幻觉

功能描述：

Agent 必须保证答案基于检索结果生成，不能让模型自由发挥。这个功能是企业知识库问答系统的核心质量要求。

子功能点：

- Prompt 中明确限制知识来源 
- 检索结果为空时，不调用或谨慎调用 LLM 
- 检索结果不足时，要求模型直接说明信息不足 
- 不允许模型编造制度、流程、数字、负责人、时间等具体信息 

 

验收标准：

- 当检索结果能回答问题时，Agent 生成答案 
- 当检索结果明显无关时，Agent 应回答“当前知识库没有足够信息回答该问题” 
- 当用户问知识库外的问题时，Agent 不应凭常识编造答案 
- 人工检查 5 个测试问题，答案必须能从引用文档块中找到依据 
- 若答案中出现具体结论，该结论必须能对应到至少一个引用来源 

建议状态：

- Q1 必须完成 

### 功能点 2.7：异常处理与兜底响应

功能描述：

Agent 需要处理检索为空、检索失败、LLM 调用失败、输出格式异常等情况，避免系统直接报错或返回不可读内容。

子功能点：

- 用户问题为空：返回参数错误 
- 检索结果为空：返回知识库暂无相关内容 
- 检索工具异常：返回检索服务暂时不可用 
- LLM 调用异常：返回模型服务暂时不可用 
- LLM 输出格式异常：保留答案文本，并返回空引用或默认引用结构 
- 所有异常都记录 trace_id 

验收标准：

- 空问题不会进入检索流程 
- 检索失败不会导致系统崩溃 
- LLM 失败不会导致系统崩溃 
- 所有异常响应都包含明确的 status 和 message 
- 日志中能根据 trace_id 定位异常发生在哪一步 

建议状态：

- Q1 必须完成 

### 功能点 2.8：基础日志与 trace_id

功能描述：

为了方便会议同步、接口联调和问题排查，Agent 层需要记录一次请求的基础链路日志。

子功能点：

- 每次请求生成 trace_id 
- 记录用户问题 
- 记录检索工具调用开始和结束 
- 记录检索返回文档块数量 
- 记录 LLM 调用开始和结束 
- 记录最终响应状态 
- 错误时记录异常信息 

验收标准：

- 每次请求日志中都能看到同一个 trace_id 串联完整流程 
- 出错时能根据 trace_id 判断错误发生在请求入口、检索、Prompt 组装、LLM 调用还是答案格式化阶段 
- 不要求接入完整链路追踪系统，Q1 阶段日志输出即可 

建议状态：

### 四、Q1 Checkpoint

Agent 层 Q1 阶段必须完成以下检查点：

- CP1：Agent 能接收用户问题，并生成本次请求的 trace_id。 
- CP2：Agent 能调用一次检索工具，获得相关文档块。 
- CP3：Agent 能将用户问题和检索文档块组装成 Prompt。 
- CP4：Agent 能调用 LLM 生成答案。 
- CP5：Agent 生成的答案必须基于检索结果，不得编造信息。 
- CP6：整个流程必须在一轮对话内完成，不实现多步循环。 
- CP7：检索为空、检索失败、模型失败时，Agent 能返回可理解的兜底结果。 

### 五、全模块验收标准

→ 组装 Prompt

→ 调用 LLM

→ 生成基于文档块的答案

→ 返回答案正文和引用来源

验收时需要满足：

- 正常问题能够返回自然语言答案   
- Agent 不会回答检索内容中不存在的信息 
- 出现异常时有明确兜底响应 
- 日志中可以通过 trace_id 追踪完整流程

## 模块 3：工具集

#### 功能描述

检索工具是 Agent 层与数据持久化层之间的桥梁。Agent 调用检索工具获取相关文档块后，组装 Prompt 发给 LLM 生成答案。检索质量直接影响最终答案的准确性和可信度。

Q1 阶段目标：实现一个可被 Agent 调用的统一检索接口，支持 **向量语义检索 + BM25 关键词检索** 的混合模式，提供结果融合排序和过滤能力，返回 Top-K 相关文档块。

#### **技术选型与接口约定**

| 组件           | 技术选型                              | 说明                                                 |
| -------------- | ------------------------------------- | ---------------------------------------------------- |
| 向量检索后端   | Milvus 2.3+ (Standalone)              | 复用数据持久化层的 Milvus 实例                       |
| Embedding 模型 | text-embedding-3-small / bge-large-zh | Q1 建议使用 OpenAI text-embedding-3-small（1536 维） |
| BM25 索引      | rank_bm25 + jieba 分词                | 基于内存的倒排索引，轻量级                           |
| 结果融合       | RRF (Reciprocal Rank Fusion)          | 无需调参，效果稳定                                   |
| 接口形式       | Python 函数 / 类方法                  | 直接由 Agent 层 import 调用                          |

**检索接口定义**：

```Python
def search(
    query: str,
    top_k: int = 5,
    mode: str = "hybrid",          # "vector" | "bm25" | "hybrid"
    filters: dict = None,           # 可选：{"doc_type": "pdf", "space": "技术文档"}
    min_score: float = 0.0          # 最低相似度阈值
) -> List[RetrievalResult]:
    """
    统一检索入口
    
    Returns:
        List[RetrievalResult]，每个结果包含:
        - doc_id: str         文档唯一标识
        - chunk_index: int    块序号
        - chunk_text: str     分块文本内容
        - score: float        融合后得分（0~1）
        - title: str          文档标题
        - vector_score: float 向量检索原始得分（mode=hybrid 时可用）
        - bm25_score: float   BM25 检索原始得分（mode=hybrid 时可用）
    """
```

#### 必须完成的功能：**详细功能点清单**

##### **功能点 3.1：检索入口与统一接口**

**功能描述**：提供统一的检索入口方法，支持三种检索模式和参数化配置。Agent 层通过该接口传入 query 和参数，获取标准化的检索结果。

**子功能点**：

- 3.1.1 实现 `SearchTool` 类，封装检索初始化逻辑（连接 Milvus、加载 BM25 索引、加载 embedding 模型）
- 3.1.2 实现 `search(query, top_k, mode, filters, min_score)` 统一入口方法
- 3.1.3 支持 `mode` 参数切换检索模式：
  - `vector`：仅向量语义检索
  - `bm25`：仅 BM25 关键词检索
  - `hybrid`：向量 + BM25 混合检索（默认）
- 3.1.4 支持 `min_score` 阈值过滤：低于阈值的文档块不返回
- 3.1.5 支持 `filters` 参数实现前置过滤（按 doc_type、space 等元数据过滤）
- 3.1.6 实现 `_validate_params()` 参数校验方法：top_k 范围 1~20，mode 必须在允许值内
- 3.1.7 返回结果按 score 降序排列

**验收标准**：

- 调用 `search("系统建模", top_k=3)` 返回不超过 3 条结果
- 每条结果包含 doc_id、chunk_index、chunk_text、score、title 五个字段
- mode="vector" 和 mode="bm25" 对同一 query 返回不同排序的结果（证明两种模式确实有差异）
- min_score=0.7 时，所有返回结果的 score ≥ 0.7
- top_k 传入 0 或负数时抛出明确异常

##### **功能点 3.2：向量语义检索**

**功能描述**：使用 embedding 模型将用户 query 转换为向量，在 Milvus 中执行 ANN（近似最近邻）检索，返回语义上最相似的文档块。

**子功能点**：

- 3.2.1 集成 embedding 模型客户端（封装 OpenAI text-embedding-3-small API 或本地 bge-large-zh 模型）
- 3.2.2 实现 `_vector_search(query, top_k, filters)` 方法
- 3.2.3 query 向量化：`embedding_model.encode(query)` → 1536 维向量
- 3.2.4 调用 Milvus `collection.search()` 执行 ANN 检索
- 3.2.5 返回结果包含：doc_id、chunk_index、chunk_text、score（内积或余弦距离）、title
- 3.2.6 支持通过 `filters` 参数构造 Milvus 查询表达式（如 `doc_id in [...]`）
- 3.2.7 实现 embedding 调用异常处理：API 超时重试 2 次，失败后抛出 `RetrievalError`
- 3.2.8 记录向量检索耗时日志

**验收标准**：

- 相同 query 多次检索返回结果一致（非随机，验证 embedding 模型确定性）
- 检索耗时 < 500ms（top_k=5，数据量 < 1000 条 chunk）
- scor**e 值在合理范围：内积模式下 > 0，且语义相近的文档 score 更高**
- **embedding API 连续失败 3 次后抛出明确异常，不阻塞主流程**

##### **功能点 3.3：BM25 关键词检索**

**功能描述**：对文档块建立 BM25 倒排索引，支持基于关键词匹配的检索。作为向量检索的补充，特别适用于精确术语匹配（如文档编号、人名、专有名词等）场景。

**子功能点**：

- 3.3.1 实现 `_build_bm25_index(chunks)` 方法：从 Milvus 中读取全部 chunk_text，建立 BM25 索引
- 3.3.2 使用 `rank_bm25` 库实现 BM25Okapi 算法
- 3.3.3 集成 `jieba` 分词器进行中文分词，处理停用词
- 3.3.4 实现 `_bm25_search(query, top_k)` 方法：对 query 分词后计算 BM25 得分
- 3.3.5 索引更新机制：提供 `rebuild_index()` 方法，在文档入库后重新构建索引
- 3.3.6 返回结果格式与向量检索一致（doc_id、chunk_index、chunk_text、score、title）
- 3.3.7 BM25 score 归一化到 0~1 区间，便于与向量得分比较
- 3.3.8 记录 BM25 检索耗时日志

**验收标准**：

- 搜索"Milvus 向量存储"能返回包含这些关键词的文档块
- BM25 结果与向量检索结果有差异化（例如精确匹配的关键词在 BM25 中排更前）
- 索引重建时间 < 2 秒（1000 条 chunk）
- BM25 score 归一化后范围在 0~1 之间
- 纯数字/英文 query 也能正常分词和检索

##### **功能点 3.4：混合检索与结果融合**

**功能描述**：将向量检索和 BM25 检索的结果进行融合排序，结合语义相关性和关键词匹配能力，提升整体检索质量。融合算法采用 RRF（Reciprocal Rank Fusion），无需手动调参。

**子功能点**：

- 3.4.1 实现 `_hybrid_search(query, top_k, filters)` 方法，并行调用向量检索和 BM25 检索
- 3.4.2 使用 RRF 算法进行结果融合：
  - 公式：`RRF_score(d) = Σ_{r∈R} 1/(k + rank_r(d))`，其中 k=60
  - R 为各检索结果排名列表的集合
- 3.4.3 去重逻辑：同一个 (doc_id, chunk_index) 出现多次时，取最高 RRF 得分
- 3.4.4 实现加权融合模式（可选）：`α × vector_score + (1-α) × bm25_score`，α 默认 0.7
- 3.4.5 最终返回按 RRF 得分降序排列的 top_k 条结果
- 3.4.6 融合后的每条结果同时保留 vector_score 和 bm25_score 原始得分（便于调试和分析）
- 3.4.7 记录融合前后的结果数量、耗时日志

**验收标准**：

- 混合检索结果与单一模式结果有差异（不完全等于其中一种的子集）
- 去重后无重复结果
- 融合耗时 < 检索总耗时的 10%
- 人工抽查 5 个 query，混合检索的相关性不低于单一模式

##### **功能点 3.5：检索质量评估与日志**

**功能描述**：为后续 RAG 质量评估和 Agent 自我提升提供数据基础。支持记录每次检索的 query、结果和评估指标。

**子功能点**：

- 3.5.1 实现检索日志记录：每次检索自动记录 query、mode、耗时、返回数量、top3 score
- 3.5.2 日志格式：`[RETRIEVAL] trace_id={id} mode={m} top_k={k} results={n} latency={t}ms top3_scores=[...]`
- 3.5.3 实现 `evaluate(query, ground_truth_doc_ids)` 评估方法
- 3.5.4 计算评估指标：
  - Hit Rate@K（K=1, 3, 5）：前 K 个结果中是否包含正确答案
  - MRR（Mean Reciprocal Rank）：第一个正确答案排名的倒数
  - NDCG@5（可选）：考虑排序质量的归一化折损累计增益
- 3.5.5 支持导出检索日志和评估结果为 JSON 文件
- 3.5.6 预留用户反馈接口：`record_feedback(query, doc_id, is_relevant)` 用于后续精调

**验收标准**：

- Hit Rate@5 ≥ 0.6（基于 20 条测试 query）
- MRR ≥ 0.4
- 评估结果可导出为 `eval_results.json` 文件
- 检索日志中可追溯每次检索的完整参数和结果

#### **功能点4.3：分块**

### 混合策略

​     有了章节树后，分块器会遍历这棵树，在切分每个章节正文的同时，把当前所处的标题路径（如 `第2章 设计方案 > 2.1 架构`）记录到 chunk 的元数据里。

- 组合“按段落”和“固定长度”：
  - 先收集一个 Section 下的所有段落。
  - 从第一个段落开始累积文本，直到累计 token 数接近上限（如 500 tokens），就在最后一个段落边界处切开。
  - 下一个 chunk 从前一个 chunk 的最后 1~2 个段落开始（重叠），保证跨越块边界的上下文不丢失。
  - 如果一个段落本身就超长，才在句子标点处强制截断，并标记截断位置。
  - 如果没超出就整段切走

### 一、模块概述

**目标**：为系统提供最小化但可验证的数据持久化能力，核心要求是**能正常写入文档块向量和文本，并能通过语义检索返回正确的文档片段**。

| 字段名        | 类型           | 说明                                         |
| ------------- | -------------- | -------------------------------------------- |
| `id`          | INT64          | 主键，自增（不暴露给业务）                   |

from pymilvus import Collection, CollectionSchema, FieldSchema, DataType

fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
    FieldSchema(name="embedding", dtype=DataType.FLOAT_VECTOR, dim=1536),
    FieldSchema(name="chunk_text", dtype=DataType.VARCHAR, max_length=65535),
    FieldSchema(name="doc_id", dtype=DataType.VARCHAR, max_length=128),
    FieldSchema(name="chunk_index", dtype=DataType.INT32),
    FieldSchema(name="source_url", dtype=DataType.VARCHAR, max_length=512)
]
schema = CollectionSchema(fields)
collection = Collection("doc_chunks", schema)
collection.create_index("embedding", {"metric_type": "IP", "index_type": "HNSW", "params": {"M": 16, "efConstruction": 200}})

def search_similar(collection, query_vector, top_k=5, doc_ids_filter=None):
    expr = None
    if doc_ids_filter:
        ids_str = ", ".join([f"'{did}'" for did in doc_ids_filter])
        expr = f"doc_id in [{ids_str}]"
    results = collection.search(
        data=[query_vector],
        anns_field="embedding",
        param={"metric_type": "IP", "params": {"nprobe": 10}},
        limit=top_k,
        expr=expr,
        output_fields=["chunk_text", "doc_id", "chunk_index"]
    )
    return results[0]  # 单向量查询的结果列表

```Python
import json, os

DOCS_DIR = "./data/documents/"

def save_document(doc_id, data):
    os.makedirs(DOCS_DIR, exist_ok=True)
    with open(os.path.join(DOCS_DIR, f"{doc_id}.json"), 'w', encoding='utf-8') as f:
        json.dump(data, f, ensure_ascii=False, indent=2)

def load_document(doc_id):
    path = os.path.join(DOCS_DIR, f"{doc_id}.json")
    if not os.path.exists(path):
        return None
    with open(path, 'r', encoding='utf-8') as f:
        return json.load(f)
```

**验收标准**：  

保存后读取内容一致；删除后 `load_document` 返回 `None`。
